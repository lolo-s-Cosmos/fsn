# fate-sandbox

实验性的互动叙事游戏：`fate-sandbox`。

大概来说就是：我想看看“角色卡”如果不再只是 prompt，而是拆成状态、工具、世界数据和几个后台 agent，会变成什么样。

当前主要在跑 Fate/strange Fake 的斯诺菲尔德，重点测的是绫香线。

它和酒馆卡的关系，与其说是平替，不如说是另一种拆法：

```txt
GM prompt
+ 世界书 / lookup
+ 结构化 state
+ 一组只能改特定领域的工具
+ 几个负责后台和审计的子代理
```

所以它会比普通聊天重很多，也更麻烦。但我想试的正是这个“重”值不值得。

项目本身也不算小。排除运行/生成目录后，大概是 `105 files / 23127 lines`。做到这个规模之后，它已经很难算“一张 prompt 卡”了，更像一个小型本地叙事 runtime。

## 和酒馆最不一样的地方

酒馆里很多东西最终都会压到聊天历史和 prompt 上：

- 角色关系靠上下文记；
- 状态靠模型自己续；
- 世界书负责提醒；
- 战斗和消耗大多靠叙事惯性；
- 时间跳过之后，后台发生了什么也主要靠模型临场补。

这当然不是说酒馆不好。酒馆的好处就是轻、直接、可塑性强。

但长线跑下来，我最受不了的是那种“模型以为已经发生了，但状态里其实没有”的幽灵事实。比如它说某个东西被捡起来了，后面又当这个东西存在；但真正要转移、丢弃、追踪的时候，你会发现根本没有这个物品。

所以 `fate-sandbox` 的思路是：能落地的东西尽量落地。

时间、地点、人物、伤势、供魔、金钱、物品、场景目标、幕后事件，都不只是写在正文里，而是进 state。正文负责把它们演出来，state 负责告诉 GM：别装，你现在到底有什么。

## 工具不是 MVU

这个也和很多酒馆迁移里的 MVU 不太一样。

MVU 更像是在模型输出后面挂一个状态栏：这轮发生了什么，然后更新变量。它关心的是 UI/变量有没有跟上剧情。

我的工具更像 GM 的行动接口。

区别大概是这样：

```txt
状态栏式：把 HP 改一下
```

而 `fate-sandbox` 更偏向：

```txt
记录一个伤势
记录一次供魔负担
开始一个 scene beat
结束一个 scene beat
记录一个玩家不知道的幕后行动
转移一个已经存在的物品
花一笔有账户、有来源的钱
```

工具的作用不是让状态好看，而是让 GM 不能随便圆。

比如 GM 想说“绫香因为宝具供魔很虚弱”，那这件事最好要么进入 condition，要么进入 servant supply / mana 后果。否则它就只是气氛。

GM 想说“幕后有人盯上了玩家”，也不能直接嘴一张。它要写成 offscreen event：谁，在什么时候，做了什么，玩家现在知不知道，会以什么痕迹投影到前台。

GM 想让一个检测仪被丢进垃圾车，也得先真的有这个 item。之前我自己就踩过坑：以为加了，结果没加，后面更新失败。这个失败其实很好，因为它暴露了模型在“假装世界连续”。

所以这套工具设计的核心和 MVU 那种“更新状态栏”不是同一件事：

> 把叙事后果变成可审计的领域事件。

这句话听起来有点工程味，但实际游玩里很直观：GM 少了一些糊弄空间。

还有一个玩下来很明显的差别：它的优势不是“工具更多”，而是“工具入口更像 GM 的行动语法”。

状态和工具当然更复杂，但入口如果设计得对，模型反而不一定更容易出错。因为它不是在自由编辑一堆变量，而是在选择一个领域事件：移动、过夜、花钱、受伤、揭示真名、推进 scene beat。参数错了，工具会打回；打回时也不是一句“格式错误”，而是告诉它当前可用的 purseId、objectiveSummary、actorId，或者告诉它当前 beat 还没结束。

这点我觉得比酒馆变量系统更适合长线。酒馆里模型写错状态，很多时候系统也会接受，只是后面剧情开始慢慢漂。这里模型也会犯错，比如空 objectiveId、钱包标签错、真名写进 public，但这些错误可以沉淀成工具约束、错误提示和测试。模型越犯错，系统边界越硬。

所以它对模型的要求不是“文笔一定最好”，而是“愿意服从工具”。只要模型会用工具，哪怕第一次参数写错，也通常能根据错误信息重试，把叙事拉回正轨。真正怕的是模型不听话：不查 lookup、不结算状态、不调用 scene / economy / condition 工具，直接开始续写。那这套系统就会被降级成普通 prompt 卡，状态和剧情重新分家。

## 子代理也不负责陪聊

我现在用了几个项目内的子代理，但它们不会和玩家直接对话。

它们更像后台导演组。

比如：

- `parallel-line`：玩家休息、赶路、潜行、过夜时，给一条幕后线候选。它不直接改状态，只是告诉主 GM：这段时间世界背面可以怎么动。
- `timeline-showrunner`：审计现在是不是跑偏了。是不是又开始悬疑化？是不是 NPC 只剩等玩家触发？是不是后台压力又变成警察巡逻和新闻播报？是不是世界 stale 了？

这部分是我最近改得最多的。

一开始 FSF 很容易被写成：警察、警察、警察。改完以后又变成：新闻、新闻、新闻。

这就说明问题不止是“少写警察线”，而是世界背面需要真的有人做事。FSF 里应该有从者、御主、教会、魔术师、土地、政府遮蔽、普通社会的错位，而不是只有“有部门监测到了异常”。

现在规则变成：后台钩子必须是可交互的。

新闻可以有，但新闻只是投影，不能顶替事件本体。巡逻也可以有，但单纯巡逻增加撑不起剧情推进。门响、信件、旧物同理：只要能归属到某个 actor / faction / location / consequence，并且给玩家调查、规避、接触、准备或撤退的窗口，就可以用；做不到的话，它就只是不可行动的气氛钩子，应该删掉或停用。

## 我最在意的也不是“写得更美”

我现在越来越觉得，长线互动叙事的问题不主要是文笔。

文笔可以调。真正难的是：

- 角色有没有自己的行动意志；
- 玩家选择有没有改变世界；
- 时间跳过后世界是不是只会等玩家；
- 代价有没有留下；
- 线索是不是能行动，而不是只负责营造气氛；
- 状态压缩以后，GM 会不会忘掉前面真正重要的偏好。

比如绫香这条线，我比较在意的是她不能退化成“被 Saber 保护的状态栏”。她可以害怕，可以缺乏经验，但她应该逐渐学会给命令、做取舍、承担 Master 的位置。

Saber 可以强，可以冷静，也可以有点坏心眼地逗她。但 Saber 不应该把所有战略都替她决定完。

这类东西如果只写在 prompt 里，很容易过几轮就漂。于是现在一部分进 prompt，一部分进工具规则，一部分交给 showrunner 审计。

## 运行方式

主要测试模型：GPT-5.5。其次跑过 Opus 4.5 和 DeepSeek V4 Pro。子代理目前预设为 DeepSeek V4 Pro，可按自己的环境改。

但比模型名字更重要的是工具调用纪律。这个项目不太适合那种“明明该调用工具却直接写正文”的模型。它可以犯参数错，工具会打回；只要它愿意按错误信息重试，状态通常能回到正轨。它如果不愿意用工具，那就会很快退化成普通聊天卡。

这是一个 pi coding agent 项目，本地跑。

大概是：

```bash
pnpm install
./start.sh
```

第一次运行需要按自己的 pi 环境配置模型或登录。

项目会把运行配置隔离在 `.pi/agent/`，游玩记录在 `sessions/`。这些是本地东西，不要拿去分享。

## 适合谁

如果你想要的是轻量开聊、角色陪伴、随便调 prompt，那这个可能太重了。

如果你对这些问题感兴趣，可能会觉得它有点意思：

- 酒馆卡能不能拆成更工程化的 agent；
- TYPE-MOON 这种设定密度很高的世界怎么做长线沙盒；
- 怎么减少“模型以为发生了但状态没有”的幽灵事实；
- 怎么让后台阵营在玩家休息时也合理行动；
- 怎么让叙事少靠气氛，多靠代价、状态和可交互后果推进。

它离完美成品还很远。更准确地说，它是我拿来试一种方向的东西：

> 角色卡如果不再只是一段 prompt，而是一套本地运行的叙事系统，会不会更适合长线游玩？
